0.  Overview

 

The Dynamic Dialogs feature in G2 8.0 provides a programmatic means of managing Microsoft Windows native dialogs from within a G2 application.

To use the Dynamic Dialogs feature the client must be running Telewindows on Windows machine using Telewindows version 8.0 of later.   The G2 server may run on any supported platform.

Dialogs are specified as a structure in G2.   Section 1 describes the details of the initial specification of a dialog in G2.

Dialogs can be posted on a specific G2 window by using the g2-ui-post-custom-dialog system procedure.  See section 2 for details on posting a dialog.   When a dialog is posted on a specific G2 window a unique dialog-id is returned by the g2-ui-post-custom-dialog system procedure.

When a dialog is updated or dismissed the G2 application receives callbacks.   Details on how the callbacks work are contained in section 3.

A dialog may be dynamically modified by a G2 application.   This is done by using the g2-ui-modify-custom-dialog system procedure.  For details on how to programmatically modify a dialog after it has been posted see section 4.

A dialog may be dynamically queried by a G2 application.  This is done using the g2-ui-query-custom-dialog-values system procedure.  For details on how to programmatically query a dialog see section 5.

There are a number of different controls supported by the Dynamic Dialogs feature.  A full reference of each control is contained in section 6.

Examples of the Dynamic Dialogs feature ship with G2 8.0.  The dialogs-demo.kb contains examples of each kind of control and a larger example that involves dynamically modifying posted dialogs and using callbacks.

2.1 Posting a Dialog

 

A dialog is posted on a remote client’s window by calling the following system procedure:

 

g2-ui-post-custom-dialog(dialog-specification: structure, user-data: value, win: g2-window) = (integer)

 

The return value of this is an integer containing the handle to the posted dialog.

If posting the dialog failed an error will be signaled by this system procedure.

The details of the dialog specification are discussed in the next section.

 

2.2 “Hello World” Dialog Example

 

Below is an example of G2 procedure code that creates a simple dialog that contains a label that says “Hello World”:

 

dialog-id =  call g2-ui-post-custom-dialog(

structure (dialog-width: 310,

  dialog-height: 124,

  dialog-is-modal: true,

  components: sequence (structure (control-type: the symbol push-button,

      control-id: 1,

      height: 14,

      width: 50,

      left: 10,

      top: 86,

      response-action: the symbol ok,

      control-value: structure (text-value: "OK")),

    structure (control-type: the symbol label,

      control-id: the symbol my-dialog-label,

      width: 50,

      height: 15,

      left: 10,

      top: 10,

      response-action: the symbol ignore,

      control-value: structure (text-value: "Hello World"))),

 the symbol hello-world-dialog,

 window);

 

 

 

3. Dialog Callbacks

 

The Dynamic Dialogs feature can be configured to send callback to G2 when controls are updated and/or when the dialog is dismissed (cancelled or Oked).

 

3.1 Response Actions

 

The response-action attribute in the dialog component structure controls

what callbacks the control will trigger when the use updates the control manually.

 

The following response-action symbols are supported:

 

 

Response Action Symbol

Description

ignore

If the control is updated do not send any callbacks

ok, accept

Dismiss the dialog and respond with all data.

cancel

If the control is updated immediately cancel the dialog and send a dialog dismissed callback (intended for pushbutton use only).

respond

If the control is updated send a dialog updated callback

respond-with-all-data

If the control is updated send a dialog updated callback including all the data of all the controls in the dialog

ok-with-just-my-data

Dismiss the dialog and respond only with the data for this control

ok-without-data

Dismiss the dialog but do not respond with any data

 

 

Note that whenever a dialog is dismissed using the windows close box this will trigger a dialog dismissed callback with dialog-was-accepted of false.

 

3.2 Dialog Update Callback

 

The dialog-update-callback attribute in a Dialog Structure configures which procedure to call when a dialog update callback needs to take place.

 

 

Callback

Description

dialog-update-callback

A procedure with signature:

 

(win: class g2-window, dialog-id: integer, control-id: value,

reason-for-callback: symbol, new-value: structure,

other-values: sequence,

user-data: value)

 

 

 

new-values is a structure that is the control-value of the control when the update takes place.

 

other-values is a sequence of control-value structures for controls other than the one for which the update takes place (initially this will be all other controls but eventually this will be configurable)

 

3.2.1 Example Update Callback Procedure

 

Below is a simple update callback procedure that just posts the results of the callback to the message board:

 

dialog-examples-update-callback(

                  win: class g2-window,

                  dialog-id: integer, control-id:      

                  value, reason-for-callback: symbol,

                  new-value: structure,

                  user-data: value,

                  other-values: sequence) = ()

begin

post "Update callback: g2-window: [the g2-window-remote-host-name of win],  dialog-id: [dialog-id], control-id: [control-id], reason-for-callback: [reason-for-callback], new-value: [new-value], user-data: [user-data], other-values: [other-values]";

end

 

3.3 Dialog Dismissed Callbacks

 

Callback

Description

dialog-dismissed-callback

A procedure with signature:

 

(win: class g2-window, dialog-id: integer, dialog-was-accepted: truth-value, user-data: value,  new-values:

sequence)

 

 

 

dialog-was-accepted:  a truth-value indicating whether the dialog was closed due to a cancel or OK button being pressed.  (see Response Actions section above for details).

4. Modifying a Dialog

After a dialog has been posted the contents of a dialog can be changed using the g2-ui-modify-custom-dialog system procedure.

 

G2-ui-modify-custom-dialog(dialog-id: integer, modify-specification: sequence, win: class g2-window) = ()

 

The dialog to be modified is specified by the dialog id.  The window the dialog is to be modified on is specified by the win argument.

The modify-specification is a sequence of structures with the following specification:

 

Attribute name

Type

Required

Default

Description

control-id

integer

Yes

N/A

Must be an id of a control that

control-action

symbol

No

replace

What kind of action to perform (See Control Actions section below)

control-value

sequence or structure

If the control-action is add or replace then this is required

N/A

The new value for the control.

 

 

 

Note that the modifications for Tabular Views and for List boxes have yet to be implemented and will likely require additional control actions.

5. Querying a Dialog

 

After a dialog has been posted the contents of a dialog can be queried using the g2-ui-query-custom-dialog-values system procedure.

 

g2-ui-query-custom-dialog-values(dialog-handle: integer, controls: sequence, win: class g2-window) = (sequence)

 

 

Argument

Type

Description

dialog-handle

integer

Must be a valid dialog handle that has been returned by g2-ui-post-custom-dialog

controls

sequence

Reserved for future use

win

class g2-window

The g2 window on which the dialog is posted.

 

 

The result of a call to g2-ui-query-custom-dialog-values will be a sequence of current control-values for all controls in the dialog.

call g2-ui-modify-custom-dialog(

  window);

Appendix J:  Dialog as Class Proposal (jv)

 

Although it's pretty late in the game, we want to add a def-class dialog object

to G2.  It would replace the top-level structure we use in the post-dialog

sysproc, although the bulk of that structure, the sequence of components, would

remain a sequence of structures.  The object would probably have the following

slots, which correspond very closely to the existing structure:

 

  • .width 

  • .height 

  • .is-modal 

  • .update-callback-procedure 

  • .dismissed-callback-procedure 

  • .user-data 

  • .components 

 

... where the bulk of the specification would be in the "components" slot.

 

The advantages I can think of off the top of my head are encapsulation, much

cleaner looking code (instead of inlining all of the text of the spec, you refer

just to the object), the opportunity to present errors as frame notes, and the

infrastructure for improvements later, like caching dialogs instead of

regenerating them each time.

 

One other "side benefit" is that we can have the components slot be edited in

the scrolling editor.  Right now, Allan and I are sometimes using free-texts and

user-defined-slots to hold the components spec, and that becomes very painful

because they open up in the non-scrolling editor.

 

The biggest disadvantage I see is just apprehension of committing us to

something that we won't easily be able to change too much in the future.  For

one thing, we won't have any grammar for the components: it will only be a

sequence of structures.  That's sort of the attribute-export way of doing

things, but we won't have a non-attribute-export way in 8.0r0.

 

(Note from meeting – the dialog as class approach is not being pursued for G2 8.0b0 or 8.0r0).
