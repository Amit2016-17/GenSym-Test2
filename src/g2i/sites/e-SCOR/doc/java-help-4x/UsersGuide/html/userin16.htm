Merging Modules

When you merge a module into your e-SCOR model, e-SCOR loads the specified module and any directly required modules that are not already open. e-SCOR does not add the merged module to the module hierarchy. Therefore, before you save your model, you must either include the merged module as a directly required module of the top-level module, or delete the merged module.

When saving the merged module with the model, you must click the Including Required Modules option on; otherwise, the merged modules will not be saved as part of the model.

When merging modules, you must consider how e-SCOR should handle conflicts if they arise. For example, if the module you are merging contains class definitions with the same name as those in the existing model, you typically want the definitions in the existing model to take precedence. By default, e-SCOR automatically resolves conflicts when merging modules by using definitions in the existing model.

When merging a module that was developed in an earlier version of e-SCOR, you might want the merged module to use the formats and system tables of the current model and e-SCOR version, thereby bringing the module up to date with the current version. By default, e-SCOR does not bring formats up to date or install system tables when merging modules.

To merge a module into an existing model:

1 Choose File > Modules > Merge to display the Merge Module dialog.
The Merge Module dialog is similar to the Open Project dialog except for the Options tab.
2 On the Path tab, navigate to the desired directory and choose the KB file to merge.
3 Click the Options tab to change the default options for merging, as follows:
Automatically resolve conflicts, which has this behavior when conflicts exist:
- When selected, the model replaces definitions of the same name in the merged module with existing definitions. To ensure that all definitions are consistent, we recommend that you select this option whenever you merge a module. This option is the default.
- When not selected, the model renames conflicting definitions in the merged module to include the module name, thereby retaining the merged module's definitions. If you believe the merged module contains newer definitions than the existing model, use this option, then integrate the merged module's definitions into the existing model's definitions and delete the merged definitions.
Bring formats up to date, which, when selected, replaces system-defined formats in the merged KB, such as the width of text boxes, with those defined in the current KB. We recommend that you not select this option when merging models developed in an earlier version of e-SCOR, unless you want to use the older formats. This option is not selected, by default.
Install system tables, which, when selected, replaces existing system table settings with the system tables in the merged KB. In general, we do not recommend that you select this option when merging models developed in earlier versions. This option is not selected, by default.
4 Click the Path tab and click the Merge button to merge the specified module.
5 Display the module hierarchy and show the properties dialog of the top-level module.
For details, see Editing Module Properties.
6 Edit the Directly Required Modules to include the merged module.
For example, if you merged a module named examples, the Directly Required Modules would look like this:
escor-customiz, examples

The module hierarchy should now include the merged module below the top-level module in the hierarchy.
