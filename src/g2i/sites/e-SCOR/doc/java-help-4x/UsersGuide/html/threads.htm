What Happens When the Simulation Runs

When you run a simulation, e-SCOR runs complex, time-multiplexed processes that execute transactions, which:

Create and delete objects such as:
- Purchase requests, purchase responses, and purchase awards.
- Orders, products, and shipments.
- Build orders, chits, and products.
Can take a specified amount of simulation time to occur.
Can have associated costs.
Compute SCOR and other performance metrics.

Each transaction that e-SCOR performs is part of a sequential process. Each process has a triggering event, which initiates the process, and a series of sequential steps, which can be triggered asynchronously. Each step occurs within a particular process category, such as the Source category, and generates a particular type of object, such as an order. In addition, each process interacts in predictable ways with the other processes.

This topic describes each process by using a diagram that describes the steps, triggering events, and objects, each of which is represented by a particular symbol. Following each process diagram is a table that summarizes each parameter and metric in each step in the process. The parameters and metrics fall into one of a number of SCOR categories, based on their function in the supply chain. Following the table is a detailed description of each process step, with links to the definitions of each parameter and metric.

Before you can understand each process, you must understand:

The process diagrams and the symbols they use.
The parameters and metrics by function.
How the processes interact.

e-SCOR has seven basic processes:

Role Verification process
Order/Product process
Build process
Financial process
Make Planning process
Source Planning process
Contracts process

e-SCOR assumes "ideal" transactions, for example, no shipments are lost, no orders are cancelled, and all invoices are paid in full and on time. When you run a simulation, every transaction completes and e-SCOR accounts for all objects and financial costs. For example, all components turn into finished products; all components and finished products are always accounted for as inventory, work in progress, or in transit amount; all components and finished products turn into invoices and payments; and all invoices and payments are always accounted for as bookings, outstanding, obligations, payments, or collections.

In general, you configure all timing parameters that advance the simulation clock, with a few minor exceptions. In particular, when the model initializes or performs its planning tasks, e-SCOR advances the simulation clock by one or two seconds, which you do not configure explicitly. These small advances of the simulation clock should not significantly affect any of the metrics that e-SCOR computes relating to processes or order performance.
