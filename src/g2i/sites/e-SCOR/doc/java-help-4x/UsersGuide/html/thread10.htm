M1 and M2 categories (Build tab)

1 Determine batches.
The M1: Make-to-Stock and M2: Make-to-Order categories determine the number of batches it needs to fill the build order by looking at the Maximum Batch Size and Minimum Batch Size parameters of the role's delivery product. It does this by breaking up the build order into separate batches, each of which is no larger than the Maximum Batch Size and no smaller than the Minimum Batch Size, which creates no more than one partial batch.
If additional build orders arrive during this step, the M1 and M2 categories fill any existing partial batches from the new build order first, before they determine new batches. For example, an additional build order might arrive, which was created to compensate for build yield. This new build order might cause the M1 and M2 category to fill an existing partial batch.
Once the M1 and M2 categories have determined the number of batches, it generates a chit, which triggers the build-order selection process.
2 Select build order.
2.1 Build-order selection triggers, based on one of two events:
The M1 and M2 categories receive a chit from the previous step, indicating that a build order needs to be manufactured.
The M1 and M2 categories receive source products as a result of the Order/Product process, described in Order/Product Process, indicating that the Make category has new components available for manufacturing.
When the build-order selection step is triggered, the M1 and M2 categories determine the Batch Preference parameter to determine whether to allow complete or partial batches.
The M1 and M2 categories might have multiple build orders competing for the available manufacturing resources. If so, it determines which build order to make first, based on the planning strategy:
For stock products, the M1: Make-to-Stock category sorts build orders, based on the Build Selection parameters, which you configure for the M1 category.
For details, see Configuring Build Selection Parameters.
For make-to-order products, the M2: Make-to-Order category sorts build orders for delivery products on a random basis. For each delivery product, it chooses complete batches before partial batches. It only selects partial batches when it has no complete batches from which to choose.
2.2 The M1 and M2 categories wait the amount of time indicated by the Order Release to Build Time parameter before proceeding to the next step.
3 Get components.
The M1 and M2 categories wait the amount of time indicated by its Production Material Cycle Time parameter.
The M1 and M2 categories apply the Production Material Handling cost parameter to its Production Material Handling costs incurred metric and to the aggregate cost metrics associated with the role.
The M1 and M2 categories attempt to retrieve the number of components from inventory that are required to manufacture the build order, as follows:
If the inventory of components is not sufficient to manufacture the build order, the M1 and M2 categories send a back order to the Make category for the portion of the order that it cannot make, which initiates Step 1. Determine batches..
If the inventory of components is sufficient to manufacture the build order, the M1 and M2 categories increment the Work in Progress metric of the source product to indicate that the role is currently manufacturing finished products, which are not yet reflected in the inventory metrics for the delivery product. It also decrements the Inventory Level and Potential Inventory Level metrics and updates the Inventory Status metric of the source product for the role. The Make category also updates these metrics for the portion of the build order that is not back ordered.
4 Manufacture products.
4.1 The M1 and M2 categories increment the Build Orders Started metric by one, then they manufacture their selected build order by creating the number of finished products in the build order that it determined in Step 2, Determine batches..
4.2 For each batch, the M1 and M2 categories wait the amount of time indicated by the Total Build Time parameter of the Make category before proceeding to the next step. The Make category takes the same amount of time to manufacture a partial batch as it takes to manufacture a complete batch.
If you configured the Total Build Time multiplier parameter of the delivery product to model differences in build times for different products, the Make category adjusts the value of the Total Build Time parameter by the multiplier.
4.3 If you configured the Yield parameter of the delivery product to model defective production in the manufacturing process, the M1 and M2 categories adjust the number of products they make by the yield; the rest is considered defective and discarded. The Make category then creates new build orders for the amount of defective product and sends the build order back to Step 2, Determine batches.. The Make category treats this build order the same way it treats any build order.
The Make category then computes the Total Products Rejected metric, based on the Yield parameter.
4.4 The M1 and M2 categories increment the Build Orders Completed metric by one. They also compute the Make Cycle Time (Days) metric to determine the performance of the manufacturing process.
5 Stage products.
Once the finished products are manufactured, the M1 and M2 categories decrement the Work in Progress metric of the source product for the role.
When the Order Type is stock or made-to-order, the Make category effectively places the finished products in inventory. It does this by incrementing the Inventory Level and Potential Inventory Level metrics of the delivery product by the number of products manufactured. It also updates the Inventory Status metric to reflect the current inventory status, which is within-threshold, below-threshold, or above-threshold.
6 Release products.
The M1 and M2 categories compute the Total Products Accepted metric of the delivery product, based on the Yield parameter of the delivery product.
The Make category releases the product, which takes the amount of time indicated by the Move to Delivery parameter, then it generates a chit, which it sends to the downstream Deliver category for delivery. The chit initiates the order-processing and fulfillment process, described in Order/Product Process.
