Intelligent Objects Development Plan

 

This document will attempt to uncover all the aspects that are needed in order to bring together a product called Intelligent Objects.  This document is divided up into three sections: Development, Training, Documentation, Quality Assurance, Packaging, and Licensing.  The content of each of these sections describes what has to be done in order to bridge this development activity to a level of a Beta Product.

Development

For the Intelligent Objects Beta version it has been describe that it will contain the following functionality:

  • .Core Optegrity Functionality: This is the current Optegrity Alpha/Beta version with the exclusion of SymCure. 

  • .Intelligent Sensor Objects: includes all sensors 

  • .Intelligent Objects: includes the Heater and Compressor objects 

  • .Configuration for Intelligent Sensor Objects and Intelligent Heater and Compressor 

 

To accomplish this list of functionality these actions must be done:

  • .Convert delivery version of Intelligent Objects to utilize G2 7.0r1 

  • .Merge the current Alpha version of Optegrity with the G2 7.0r1 version of Intelligent Objects. 

  • .Provide piping from the Intelligent Object to the messaging component within Optegrity and/or SymCure 

  • .Re-organize the class hierarchy to where the Intelligent Objects have a base class of Optegrity classes and/or grtl-domain-object.  This will give the Intelligent Object the capability of sending and receiving events. 

  • .Event callbacks must be created to handle events thrown from the Intelligent Objects 

  • .Event callbacks must be created to handle events thrown by the Optegrity compoents, which include but are not limited to GEVM, GEDP, GDSM, and CDG. 

  • .Modify the Intelligent Sensor to accept data from GDSM. 

 

Training

For long term goals, training will have to devise two training plans for the Intelligent Objects.  Each of these training plans must optionally include training for Optegrity, with SymCure being an addition as well.

 

  • .Customer Training: General training covering all of the Intelligent Objects.  This should include at a minimum the installation, configuration, and responding to events of the Intelligent Objects. 

    • o.General coverage training 

    • o.Intelligent Object specific training 

  • .Consulting Training: Customer Training plus the training on creating and deploying new Intelligent Objects. 

 

The Consulting Training can later be used to train Customers (i.e. OEM’s, etc.) who wish to build their own Intelligent Objects.

Documentation

Below is a list of documents that must be created in order to provide the Customer with enough information to install the product, configure the product, and operate the product.  Additional document(s) may be created at a later time to include the creation and deployment of Intelligent Objects.

  • .User’s Guide: Covers usage of Intelligent Objects and Intelligent Sensors 

    • o.Currently in edit; however, needs to include coverage of Intelligent Objects and not just Intelligent Sensors 

  • .Configuration Guide: Describes the configuration of each Intelligent Sensor and Intelligent Object 

    • o.Currently in edit 

  • .Intelligent Object Developer’s Guide: Describes how to create a new Intelligent Object and deploy it to a working system 

Quality Assurance

Quality Assurance will need to create a base line assessment of the Intelligent Objects product.  This can be accomplished by taking the delivered Intelligent Object software, as delivered by Key Control, and create test scripts to obtain base line test results.  These results can then be used as expected values to be compared against results taken after the merger of the Intelligent Objects and Optegrity.  Additional tests will have to be created to account for the functionality provided by Optegrity and used by the Intelligent Objects.

 

Below is a list of things that needs to be done in order to achieve the plan stated above:

  • .Base-line Intelligent Object code running under G2 7.0r1 

  • .Data files 

    • o.Actual data from Customer 

    • o.Conjured data 

  • .Devise test plans for existing Intelligent Object functionality 

  • .Devise test plans for added functionality provided by Optegrity 

Packaging

The Intelligent Object product will consist of the Optegrity core product and additionally the Intelligent Objects software.  Intelligent Objects software contains both the Intelligent Sensors and Intelligent Objects.  Packaging may include the division and/or inclusion of both of these capabilities.

 

Order Services must have the capability to generate license keys based on the number of Intelligent Sensors and Intelligent Objects.

 

A Customer Support Engineer must be identified to handle all Issues, Bugs, and Suggestions related to Intelligent Objects and Richard Anderson must be identified as the First Line Developer.

 

HelpDesk must be populated with the Intelligent Objects product name and appropriate versions must be added to account for Alpha, Beta, and release versions.

 

Licensing

Licensing must be handled by limiting the Customer to a specified number of Intelligent Sensors and Intelligent Objects.  Currently there is not a way to limit the number of objects through the standard way Gensym license products.  We will have to rely on the licensing scheme that was devised by Key Controls.  This is comprised of a procedure that is run at start-up and at predetermined intervals.
