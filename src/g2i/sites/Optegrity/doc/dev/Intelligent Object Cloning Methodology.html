Intelligent Object Cloning Methodology

 

Cloning an Intelligent Object will require extensive object specific programming.  Some aspects of an existing object will be difficult, if not impossible, to duplicate exactly.

 

The question of whether clone should even be allowed should be discussed thoroughly before any development should be started.  Critical issues include:

Many relationships between an Object and related Objects are 1 to 1.  For instance, an Intelligent Flow indicator can be the feed flow of only one Intelligent Heater Object.

What should be done about events that use custom message objects?  There is no way to guarantee that message content of custom message objects can be sufficiently modified for a cloned object.

We cannot assume that a user has left sub-Objects and event attribute objects on the appropriate sub-WS prior to cloning.  Therefore every sub-Object and event attribute object must be independently recreated.

 

The primary tasks that must be performed for cloning an Object are:

 

Identify the original cloned Object and extract basic properties.

Delete all objects on the new Object sub-WS.

Block new Object event processing.

Request and validate the new Object name.

Build, name, and relate any sub-Objects.

Clone related event attribute objects from original Object, place on appropriate new Object sub-WS, and, create required relationships, and update event scheduler lists.*
