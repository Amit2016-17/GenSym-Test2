Displaying a Trend from the Message Browser

 

This document will outline the basic requirements to provide additional functionality to the message browser implemented in the G2i architecture.  This requirement originated from ERG Petroli, who is currently using the Optegrity and Intelligent Objects product.  In the original Key Control Intelligent Object, there was a capability of displaying up to four plots within a trend.  This trend was accessible through the Key Control message browser when a message was selected.  The specification for the trend was predetermined and configured in the ESIS Rule Object (ERO).  The specification included a text string with certain text strings representing different options for the trend.  Below is an excerpt from the Key Control documentation explaining the design of the trend capability:

 

Trend-Chart

 

The Trend-Chart attribute is a text string used to associated up to four (4) values for trending to be associated with an ERO message.

 

The Trend-Chart attribute is accessed by the Trend Button on an ESIS Interface Frame.  Refer to the ESIS Interface Frame section of this document for more details on the Trend Button functionality.

 

Up to four variables or parameters can be specified for a trend, separated by commas.  A sample trend specification would be:

 

“the pv of F101, the sp of F101, the op of F101, the pv of T102”

 

There are three optional parameters that can be specified for each trended variable.  Each optional parameter is identified by using a specific character string in the trend specification.

 

The optional parameters are:

 

Use local trend object history <>h

Significant digits   <>d

Title for trended variable  <>t

 

<>h This code tells the trend object to use local trend object history instead the history stored by the variable.  When using this code, you will see no historical values when the trend is initialized.  Rather, the trend object collects history as new current values become available.

 

This setting is normally used for parameters (that do not keep history) as opposed to variables (like the pv of and indicator), such as a high limit.

 

<>d This code is always followed by a number signifying the number of significant digits to be displayed (<>8, <>11, etc).

 

The value of some variables, like heat duty, may normally have values with several significant digits.  The default trend settings will convert these values (1,450,000.0) to scientific notation (1.45e6).  This can be confusing to read.  By setting the <>d specification to high value, the trend will display the value without scientific notation.

 

<>t This code is followed by a text string that will be displayed as a description for the displayed variable in the trend:  For example, <>t Outlet Temperature.

 

The order of the optional parameters does not matter, but they must follow after the trend variable specification.  The following example uses all special codes:

 

“the pv of f101 <>d8 <>t Flow PV, the sp of f101 <>8 <>t Flow SP, the op of f101 <>t Controller Output, the hi_limit of the pv of f101, <>h <>d8 <>t High Flow Limit”

 

 

In this specification, 4 trends were configured as follows:

 

Trend 1:

trended value:  the pv of f101

history:  variable history

significant digits : 8

title   Flow PV

 

Trend 2:

trended value:  the sp of f101

history:  variable history

significant digits : 8

title   Flow SP

 

 

Trend 3:

trended value:  the pv of f101

history:  variable history

significant digits : default

title   Controller Output

 

 

Trend 4:

trended value:  the hi_limit of the pv of f101 (not a variable!)

history:  local trend history

significant digits : 8

title   High Flow Limit

With the above design already in place, we need to provide a conversion of the current trend specification to a specification we implement within Optegrity.

 

Since the trend specification will involve the G2i GEVM module it would be beneficial to extend this capability beyond just trends.

 

In order to provide the same functionality, an attribute must be exist in the GEVM class to contain this specification.  The following structure is proposed to capture a single plot item within the current trend information in the ERO object:

Structure (plot-item: <item-reference>,

 history-item: <history-item-reference>,

 significant-digits: <numeric value>,

 plot-item-title: “Title text”)

 

A sequence with up to four structures could be specified at the message level and stored within the new attribute of the GEVM class.

 

To provide a capability beyond the trend-only specification, I propose the following to be stored within the new attribute of the GEVM class:

 Sequence (structure (display-type: <trend-spec or workspace-spec>,

                                              Display-spec: sequence))

Where the sequence value for the display-spec would be a sequence of plot-items for the trend-spec or a sequence of kb-workspace references for the workspace-spec.

 

From the Message Browser a button would exist to act upon the information provided in the new GEVM attribute.  The button should be enabled/disabled based on the existence of information in the attribute.  The work-procedure behind the button should determine which action to take depending on the display-type and process the display-spec accordingly.
