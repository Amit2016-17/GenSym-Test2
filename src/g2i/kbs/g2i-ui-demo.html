MILL DEMO / G2i Native User Interface Demo

Background

 

This Mill knowledge base is intended as a demonstration example that illustrates several of G2’s powerful capabilities for applying expertise in real time and for rapid development as well as demonstrate a framework to build native user interfaces and how to extend it.  This is an example that was built for demonstration purposes and should not be considered representative of a fully deployed G2 application. 

 

The scenario is a metal milling operation, which in this case is for manufacturing of bolts from metal forgings.  This application models and simulates the flow of work-in-process raw materials as they are processed through various workstations including distribution stations, conveyers, machine tools and surface treatment stations.  Through the use of G2’s object modeling, rules, and procedures, the work-in-process is efficiently moved through the network of workstations and performance metrics for the operation are determined and displayed.  This example is a model with simulated data; however, a similar, deployable example could just as easily be connected to live production data to represent the actual state of the operation and to apply expertise for monitoring and controlling production. 

 

Key G2 Capabilities to Explore

 

Here are a few of G2’s capabilities to explore through the Mill Demonstration:

Object Modeling

Object modeling – G2 expressions understand when objects are related to each other, such as the individual workstations in this example that are connected via graphical connections.  Through this understanding, logic can be represented that applies generically across networks of connected objects.  This is extremely valuable for modeling complex operational systems.  To illustrate this power, try cloning all of the equipment in one of the production lines, connect them together, connect the group to the discrete-event-splitter on the left side of the Warehouse workspace and then to the finished goods warehouse object on the right side of the workspace.  Then restart G2 to reset and begin the application.  If you have connected everything correctly, the new line should just start working. 

Object Orientation

Object orientation – G2 is an object-oriented environment that takes full advantage of the reusability of object properties and behaviors.  G2’s objects are graphical, which adds to their ease-of-use.  For example, create a new machining production line from scratch by opening up the machine palette; then cloning some machines on the warehouse workspace; and connecting them together and to the discrete-event-splitter and to the warehouse object.  Restart G2 to see the new line producing product.

Animation

Animation – G2 readily enables animations of objects, such as the movement of materials in the Mill demonstration.   The Gantt chart in the demonstration is also an example of an animation.  Each object image can also be animated through manipulation of layers and through formulas that dynamically change dimensions.  The bolts and the drilling stations are examples of this type of animation.  The animations are valuable to communicate the state of an operation to both end users and to application builders.  To move the animation from real-time mode to simulation mode, change the scheduler mode (found in system tables / timing parameters) from “real time” to “as fast as possible”. 

Rule and Procedure Based Logic

Rule and procedure based logic – G2’s powerful rule and procedural languages represent application logic using an intuitive structured natural language.  The language offers a rich set of expressions for reasoning about objects, their relationships, and about time-based events.  This application particularly takes advantage of the language’s capabilities to represent the coordination logic needed to rout the movement of materials through a network of workstations.  G2 is often used for these types of coordination applications.  To examine samples of the rules and procedures in this application, go to the Mill Top Level workspace and then navigate to the workspaces for Mill Rules and Formulas and Mill Procedures. 

 

Try experimenting by writing your own simple rules to monitor the milling operation.  For example, on a new workspace create the following rule:

 

Whenever the total-part-processed of any mfg-equipment receives a value and when the total-part processed of the mfg-equipment > 50000 then inform the operator for the next 20 seconds that "The [the name of the mfg-equipment] has processed [the value of the total-part-processed of the mfg-equipment ] parts.  Please schedule preventative maintenance as soon as possible.”   

 

This rule is, obviously, very simple for illustration purposes.  However it shows several powerful capabilities of G2 rules including:

 

  • Understandability by non-programmers;
  • Working generically across entire object classes (in this case “any mfg-equipment” which includes all of the different workstation subclasses);
  • Understanding time-based events (“Whenever … receives a value…”); and
  • Representation of time (…for the next 20 seconds”). 

Data Visualization and Inferencing

Data visualization and inferencing – G2 has built in trend charts for visualizing data.  For example, open the Mill Statistics workspace to see two configured trend charts that display the machine capacity utilization, percent of run time and the percent of nodes on line.  Feel free to experiment by editing the properties of these charts. 

 

Notice also that these statistics are calculated in real time based on data that comes from sensors.  G2 is an excellent environment for deriving these types of statistics and operational performance measurements in real time.  G2 is also excellent for inferring real-time conditions about a process that are not readily measurable with physical devices or easily calculated using formulas.  For instance, G2 might infer whether the production operation is running with high, medium, or low efficiency based on assessment of several factors.  This example does not directly illustrate this capability; however, try creating your own rules and/or procedures that infer these types of conditions about this milling operation.

Native User Interface

This example leverages a predefined native user interface and demonstrates with very simple examples how to extend it. The basic user interface is implemented in the GRTL kb module with the exception of message browser, which is implemented in the GEVM module. It demonstrates how to customize the menu bar, toolbars, popup menus, dialogs and window panes such as tree views, shortcut panes and html views.
