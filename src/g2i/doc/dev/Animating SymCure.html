Dynamically updating SymCure’s specific fault model

 

SymCure now allows you the option to build a complete specific fault model associated with a domain object before any event is generated.   Initially, every event in the model is “unknown” (i.e., blue).  SymCure automatically builds and displays a view of the specific events associated with each diagnosis manager on separate workspaces.   These workspaces are updated dynamically, whenever there is any change in the underlying events.

 

How to access this feature.

  1. 1.In the config.txt file, set CDG-DISPLAY-ANIMATED-SPECIFIC-FAULT-MODEL=true.  Also set CDG-DIAGNOSIS-DELETION-MONITOR-INTERVAL and CDG-INCREMENTAL-DIAGNOSIS-MONITOR-INTERVAL to be negative numbers.  This disables automatic diagnosis deletion and incremental model construction. 

  2. 2.Choose any domain object for which you want to build the specific fault model.  (The specific fault model may contain correlated events on other objects; choosing an object is simply a way to control the scope and size of the specific fault model.  If you want every possible event, choose the menu option on the process map.)  Select the user-menu-choice initialize-specific-model.  This builds a complete specific model that incorporates every event that can be generated for the domain object.  For example, in fo2demo on initializing specific model for T_102, we get the following complete specific fault model.

Notes:

  1. 1.SymCure must be offline while the complete specific fault model is being constructed (Step 2).   Generating the complete model requires creating specific events and this may interfere with an ongoing diagnosis. 

  2. 2.This feature isn’t set up to be automatic.  This allows the user to choose which object to select for building a specific fault model.  It is possible to automate this with the use of APIs.  You can also specify the construction of specific models to include a sequence of objects or the complete process map. 

  3. 3.It is technically possible to let the user manually select a set of objects for displaying specific models through the user menu choice.  However, SymCure must be offline during this sequence of operations.  If this is desired, we will provide a switch that toggles SymCure from offline to online and vice versa.   

  4. 4.This feature must be used with great care because the specific model display can get difficult to view once the number of events increases.  Also, in order for this to work you must disable SymCure’s mechanisms for deleting specific events that are no longer relevant for diagnosis.  This means that all events will stay in memory indefinitely.  (Note: this is not a memory leak, because SymCure will reuse the same events as needed.) 

  5. 5.You can hide the view of the specific fault model and access it again through the menu choice Modules->Fault models->Specific fault models.  (Be careful not to delete it for now, otherwise you will lose the dynamic updating capability.) 

  6. 6.Animation works for this view precisely because we’ve built the entire specific model.  (The main problem with animating a view of the specific fault model is that by design for the sake of efficient performance, SymCure does not build the entire model.  Animating a view is difficult because it is impossible to know in advance how to layout the events in the fault model when it is built incrementally.) 

  7. 7.We recommend that once you have established confidence in the application and its knowledge base, for the sake of efficient deployment, you do not animate the specific model.
