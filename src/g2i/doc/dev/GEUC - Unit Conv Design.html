1.0        Introduction

This document provides the specifications for creating a Process Unit Conversion module that performs process unit conversions within an Optegrity application.  The purpose of the Process Unit Conversion module is to provide a generic methodology for handling the various units of measure used by Gensym’s international clientele.

The primary objective of the process unit conversion module is to the creation of generic process based modules and applications that can be reused by many clients regardless of the local process units standards.  It is expected that the primary users of the Process Unit Conversion module will be application consultants from Gensym and Gensym’s Partner companies and end-user application developers. And should specify used by Intelligent Objects

Process unit conversions are critical to creating truly generic first principle based models for a process plant.  Operations personnel need to be able to see the results of engineering calculations and process performance parameters in the process units to which they are accustomed so that sound decisions about process performance can be made.  Therefore, a comprehensive methodology for applying local unit conversion to process model inputs and results is mandatory for successfully penetrating the process industry market.

2.0        Functional Design Specifications

Process unit conversion from one set of units to another is very simple from a computational point of view.  Unfortunately, the wide variety of different measurement units that can be used for any particular dimension, combined with the many alternate spelling and contractions used for a given process unit, makes unit conversion a complex task.  A process unit conversion system must be able to handle a wide variety of field measurement units and unit spelling variations.

Process measurements typically belong to one of two unit systems, English or Metric (SI).  Unfortunately, there are multiple units of measurement for some dimensions in each of these systems.  For instance, users of the metric system may use pressure measurements in bar, kilogram per centimeter squared (kg/cm2), or the official standard units of Pascals.  Furthermore, field measurements are typically scaled so that a board operator does see more than five (5) digits in front of a decimal point.  For instance, small volumetric flows in a process unit may be reported of cubic feet per hour (ft3/hr) where as large volumetric flows in the same process unit may be reported as million cubic feet per day (MMft3/day).  The impact of this variety of choices is that process unit conversion cannot be performed simply on the basis of whether the end-user’s process field instrumentation is based on the English or Metric system.  Unit conversion must be made based on the field units of a given measurement and the required input units of an engineering calculation.

2.1        Unit Conversion Methodology

As stated earlier, unit conversion calculations are very straightforward, requiring only multiplier and offset coefficients.  The design challenge for unit conversion is to create an efficient methodology that can be applied to a wide number of potential unit conversion choices while keeping it simple for developers and end-users to add new conversions that are not already part of the Process Unit Conversion module.

An additional concern for the module is to highlight missing unit conversion definitions as they are requested by engineering calculations.  This will assist the developers and end-users with any necessary customizations.

3.0        Detailed Design Specifications

The conversion of process units will be performed through a single API method supported by pre-defined and user-defined unit conversion objects.  A combination of organization and execution techniques will be used to allow maximum flexibility for the application developer while executing each unit conversion as efficiently as possible.

3.1        Unit Conversion Methodology

The methodology for process unit conversion can be separated into two functionalities, the calculation for performing a specific unit conversion and the efficient processing of a set of unit conversions for a specific application.

3.1.1        Unit Conversion Equations

Process unit conversions are always a simple linear equation in the form of:

y = Ax + B Multiply First(1)

or

y = A(x + B)Offset First(2)

where:

A  - Conversion multiplier coefficient (y units / x units)

B  - Conversion offset coefficient (x units)

x  - Process unit to be converted

y  - Converted process unit

For unit conversions where either the multiplier coefficient equals 1.0 or the offset coefficient equals 0.0, either equation 1 or 2 can be used for conversion.

For unit conversions where multiplier coefficient does not equal 1.0 and the offset coefficient does not equal 0.0, the order of the conversion calculation is specific and the appropriate equation (1 or 2) must be selected for a given unit conversion.  For instance, conversion of temperature from degrees C to degrees F must use equation 1, whereas temperature conversion from degrees F to degrees C would require equation 2.

Unit conversion objects, however, are intended to be usable in both conversion directions.  A unit conversion object defined to convert degrees C to degrees F will also be used to convert degrees F to degrees C.  For reverse conversions, the inverses of equations 1 and 2 must be used:

Equation 1 inverse is:

y = (x – B) / A(3)

Equation 2 inverse is:

y = (x / A) - B(4)

A newly created conversion object should be inserted into the Unused object list described in section 3.1.3.3.1.

3.1.2.4        Cloning an Existing Conversion Object

Cloning of an existing unit conversion object should not be permitted.  By definition, each unit conversion object must be unique.  It is also not possible to have multiple conversions for a given set of process units.  Therefore, cloning an existing unit conversion object would most likely result in conflicting unit conversion objects, and the user would be force to either delete the object or change almost every critical attribute.  The unit conversion manager should prohibit the cloning of an existing object.

Cut, copy, paste, cloning should all be supported in anything we do and users should not be restricted by such a functionality. The question should not be do we support it or not, but what are the policies that we should implement after a paste or a clone. For example reset any attributes that need to be filled out by the user and cannot be kept during a paste/clone because duplicate items exist

3.1.3.1        Conversion Request Method

A unit conversion request will be made through a method call in the following format:

y, status = f(x, dimension, input units, output units)        (5)

where:

y    - Converted process value, float

status   - Conversion status, symbol

x    - Process value to be converted, float

dimension  - dimension to be converted, symbol

input units  - input x process units, text

output units - - output y process units, text

This method will locate the appropriate unit conversion object, determine whether the conversion is direct or inverse, and perform the specified conversion equation (1, 2, 3, or 4) using the supplied coefficients.

Once a unit conversion object with the requested input and output process units has been located, the following logic should be used to select the appropriate conversion equation.

when the equation type of the unit conversion object is Multiply First:

and the requested input units

match one of the unit conversion object input units

then use Equation (1) for the conversion

otherwise use Equation (3) for the conversion

when the equation type of the unit conversion object is Offset First:

and the requested input units

match one of the unit conversion object input units

then use Equation (2) for the conversion

otherwise use Equation (4) for the conversion

In addition to performing the requested unit conversion, this method will also update the execution data for the unit conversion object

NOTE: The specifications and calculations for the execution data attributes are to be defined for the final version of this document.

The status returned by the unit conversion method should communicate any problems encountered during the execution of the desired unit conversion.  The following probable status codes are suggested: - why not throw/signal exceptions which ensures it has to be dealt with

3.1.3.2        Conversion Object Request Matching

Users often have different methods of referring to common process units within their DCS and Process Information systems.  For instance, temperature may be labeled as “DEG F” or “DEGF”.  A flow may be labeled as “M3/H” or “m3/h” or “M3 / H” or “m3 / h”.  Often, multiple versions will be used within a single DCS or Process Information system.  Therefore, all input and output request and unit conversion object input and output specifications will be stripped of spaces and converted to all capital letters during the search process to minimize the number of unit alternate unit spellings that will need to be defined for a given unit conversion object.  This will allow a unit request for “m3 / h” to be matched with a unit conversion object defined to convert “m3/h” as both unit specifications would be converted to “M3/H” during run-time execution.

3.1.3.3        Conversion Object Organization

It is anticipated that over time that there will easily be hundreds of potential unit conversion objects that can be accessed by an application developer.  Therefore, and efficient method of finding a specific unit conversion object during run-time is critical.  One method of improving the efficiency of finding a specific unit conversion object is to organize all unit conversion objects into object lists.  These object lists are to be based on the dimension attribute of each unit conversion object.

Maybe data structure  we are looking at adding in g2 could help here and improve performance

This document will frequently refer to object organization using the term “list”.  The term list is used for descriptive purposes and is not intended to imply that any specific programming methodology should be used to achieve the objectives of this design document.

There are to be three varieties of object lists used to organize the unit conversion objects:

  • .Unused unit conversion object list 

  • .Missing unit conversion object list 

  • .Accessed unit conversion objects lists 

3.1.3.3.1        Unused Object List

During application initialization, all existing unit conversion objects are to be placed into an Unused object list.  When a specific unit conversion object has been requested by the user via a procedure call, the object will be removed from the Unused object list and placed into one of the Accessed object lists.

3.1.3.3.2        Missing Object List

The Missing object list is initially empty.  Whenever a specific unit conversion has been requested, but no matching unit conversion object can be found, a default object will be created, the input and output values set for the requested unit conversion, and placed into the Missing list.  The existence of one or more default objects will be highlighted to the application developer so that the unit conversion API request can be modified or the newly created unit conversion object properties can be specified.

If the user edits the new unit conversion object, a successful exit from the edit dialog should include moving the new object into the Unused object list.

3.1.3.3.3        Accessed Object Lists

There will be Accessed object lists for each of the packaged dimensions listed in Section 2.2.  A miscellaneous Accessed object list will be built for any dimension not listed in Section 2.2.  When a unit conversion request requires a unit conversion object currently residing in the Unused object list, it will be moved to the appropriate Accessed object list based on the dimension attribute of the unit conversion object.

3.1.3.3.4        Object List Search Technique

When a user unit conversion request is made, the appropriate unit conversion object will be located using the following search technique:

  1. 1.The dimension specific Accessed object list 

  2. 2.The Missing object list 

  3. 3.The Unused object list 

Each unit conversion object in an object list should be checked against the desired input and output units for both conversion directions so that it is not necessary for the developer to define a conversion and its inverse.  For instance, if a request is made to convert degrees F to degrees C, the applicable unit conversion object may be specified in the orientation of converting degrees C to degrees F.  Therefore, each unit conversion object should be checked to see if the inverse of the specified unit conversion will provide the requested conversion.

If the unit conversion object that matches the requested units is currently in the Missing object list, then the API method should return an unsuccessful status value.

This search technique should, within a few hours of application initialization, move most, if not all, of the necessary unit conversion objects for the application into short, dimension specific object lists.  While initial unit conversion requests may take some time to process, subsequent requests will be executed very efficiently.

3.1.3.4        Conversion Request Frequency

As stated previously, the search for the appropriate unit conversion object for a requested unit conversion is somewhat cumbersome.  As a further refinement to the search technique specified in Section 3.1.3.4, the order of the unit conversion objects within the dimension specific Accessed object lists will be updated based on the frequency of conversion requests.  This will improve unit conversion search efficiency by checking the most heavily used unit conversion objects within a dimension specific Accessed object list first.

Each dimension specific Accessed object list is to be periodically scanned for the need to re-order the unit conversion objects.  A time-based algorithm is to be used to initiate the potential re-ordering of Accessed object lists.  The algorithm will initiate the re-ordering of the Accessed object list based upon the time since application startup.  The need for re-ordering the Accessed object list should decrease as the time since application startup grows, since it is expected that the access order for each list will stabilize within a few hours of application startup.

3.2        Packaged Unit Conversions

The Process Unit Conversion module should supply the developer with pre-defined unit conversion objects for the most commonly used process unit conversions.  This will enhance the development capabilities of Optegrity users by minimizing the number of custom unit conversion objects that need to be built for a new application and eliminating the inevitable configuration mistakes and the subsequent time spent debugging unit conversion problems.

3.2.1        Packaged Definitions

Pre-built unit conversions are to be defined for typical process units.  The unit conversion data specifications, including equation type and conversion coefficients will be listed in dimension specific appendices included with this design document in Excel spreadsheets.  The spreadsheet appendices should be updated with new unit conversion definitions as application developers find the need for additional standard unit conversions or new alternate spellings for existing unit conversion objects.

 

  • .Length 

  • .Area 

  • .Volume 

  • .Mass 

  • .Volumetric Flow 

  • .Mass Flow 

  • .Pressure 

  • .Temperature 

  • .Enthalpy 

  • .Heat Capacity 

  • .Energy 

  • .Power 

The challenge for packaging pre-defined unit conversion objects is that there are many ways to textually refer to some units of measure in Distributed Control Systems (DCS).  In many cases, the end-user is going to automatically read in process units from their DCS system.  It is not reasonable for Gensym to require application developer to mandate specific spellings for a given set of process units.  Therefore, pre-built unit conversion objects will be defined with as many different spellings as possible to support a variety of end-user standards.

3.2.2        Definition Loading Files

The initial set of defined unit conversions, including the definition of possible alternate spellings, will likely mean that several hundred unit conversion definitions will need to be built for the initial release of the Process Unit Conversion module.  Therefore, a method for loading the object attribute definitions from a flat file is to be created.  Likewise, the capability for recreating a loading file from the development environment is to be provided.

In the interest of easy upgrade and expansion capabilities, it will be desirable to build unit conversion objects dynamically for an application from a Gensym managed data file during application initialization.  User customizations for a specific application should be automatically stored into an external custom definition file as they are interactively defined through the unit conversion object manager.  The custom definition file would be re-loaded during application initialization after the loading of the Gensym standard definition file.  A user custom definition file would also make it easy for an end-user to feed back to Gensym new unit conversion definitions as well as new alternate unit spellings for exiting unit conversions.
