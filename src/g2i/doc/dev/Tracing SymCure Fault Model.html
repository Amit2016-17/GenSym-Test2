Tracing SymCure’s event propagation algorithm over Specific Fault Models

 

SymCure provides you with a new powerful feature that allows you to step through the propagation and construction of events in a specific fault model.   This feature will help you to understand the behavior of SymCure’s event propagation algorithm and the propagation logic of the different events.  Plus, it will be very helpful in debugging the fault models for your application.  

 

To access this feature, you must enable SymCure’s debugging mode.  You can do this by initializing the parameter CDG-DEBUG-MODE in config.txt to true.  (Ideally, you would set CDG-DEBUG-MODE to true during the application development stage.  At deployment, CDG-DEBUG-MODE should be set to false.)

 

When the debug mode is enabled, SymCure creates a log of its event propagations.  The new feature allows you to graphically step through this log.  You can also save this log as an XML file (partially implemented --- needs UI for access and some other refinements).   The user interface for tracing the specific fault model step by step consists of a GUIDE dialog panel that controls the debugging process, and a KB-workspace which shows the graphical trace.

 

You can call up the debugging panel from the Optegrity menu (to be implemented --- for now load cdgq, go to Manual Regression Tests->API tests->Incremental debugging and press the button for start cdggrlb-debug-session).  The debugging panel is shown below.

 

 

SymCure’s event propagation algorithm is driven by incoming events, i.e., event propagation commences only after the arrival of an incoming event.  At any time during the trace, the panel displays the latest incoming event (Row 3).  The event propagation algorithm is a best first search algorithm that traverses through the specific fault model to compute values for each of its events.  The latest event to have its value computed, i.e., Current Event, is displayed in Row 1.   The event that led to Current Event, i.e., Previous Event is displayed in Row 2.

 

The status of the panel contains the following symbols

  • -.READY, i.e., the debugging process is ready to be started 

  • -.IN-PROGRESS, i.e., the debugging process is in progress 

  • -.END-OF-INCOMING-EVENT, i.e., you have reached the end of propagation for the latest incoming event 

  • -.END-OF-LOG, i.e., the end of the log has been reached and the debugging session is complete. 

 

You can step through the fault model with the help of this table.  However, it is far more appealing to step through the fault model graphically.  When the debug panel is invoked, SymCure also shows the graphical debug display workspace (which is initially empty).  (Perhaps the debug display workspace should be visible only upon user command --- see View Graph below?)  SymCure constructs and propagates events step by step on this workspace.

 

The following buttons control the trace process.

  • -.Next Step: moves the trace process forward by one step 

  • -.View Graph: shows the graphical display workspace 

  • -.Hide Graph: hides the graphical display workspace 

  • -.End Session: ends the debugging session, closes the panel, and deletes the display workspace (note: this is the only safe way to delete the display workspace) 

 

 

You can see the debug panel and the corresponding graphical display workspace in the figure above.  This is constructed one event at a time by pressing Next Step.  The last edge traversed by the algorithm is shown in red.  You can view any of Current Event, Previous Event, or Incoming Event by selecting the appropriate radio button and pressing View Event.  A red arrow points to the selected event on the display workspace.  

 

The following menu choices are associated with the specific event display objects.

  • -.Properties 

  • -.Show Target (not sure if this is necessary, what if target is removed before debug session?) 

  • -.Go To Generic Event 

  • -.Show Event History 

 

Notes:

  1. 1.The graphical debugging feature does not refer to its underlying specific events in any way.  It is constructed solely from a log of the fault model, although it does require access to the generic fault models and the target objects.  Thus, you can debug the log of the fault model at any time including even after the specific fault model has been deleted.  

  2. 2.Each debugging session uses a local copy of the log of the specific events.  Thus, it is possible (in theory at least, to be tested) to support multiple simultaneous debugging sessions.  

  3. 3.If at deployment time, you want to dynamically enable or disable the debugging mode, SymCure provides an API cdg-set-debug-mode(Mode: truth-value).  When mode is true, the debug mode is enabled.  Any subsequent events processed by SymCure will be logged and therefore available for graphical debugging.  When Mode is false, the debug mode is disabled and the event log is emptied.
