1.0        Introduction

This document provides the specifications for developing an Engineering Unit Conversion module that performs engineering unit conversions within an Optegrity application.  The purpose of the Engineering Unit Conversion module is to provide a generic methodology for handling the various units of measure used by Gensym’s international clientele.

The primary objective of the engineering unit conversion module is the creation of generic process based modules and applications that can be reused by many clients regardless of the local engineering units standards.  It is expected that the primary users of the Engineering Unit Conversion module will be application consultants from Gensym and Gensym’s Partner companies and end-user application developers.  It is also expected that all Intelligent Object software will use this module to generically support the use of multiple unit systems and end-user languages.

Engineering unit conversions are critical to creating truly generic first principle based models for a process plant.  Operations personnel need to be able to see the results of engineering calculations and process performance parameters in the engineering units to which they are accustomed so that sound decisions about process performance can be made.  Therefore, a comprehensive methodology for applying local unit conversion to process model inputs and results is mandatory for successfully penetrating the process industry market.

2.0        Functional Design Specifications

Engineering unit conversion from one set of units to another is very simple from a computational point of view.  Unfortunately, the wide variety of different measurement units that can be used for any particular dimension, combined with the many alternate spelling and contractions used for a given process unit, makes unit conversion a complex task.  An engineering unit conversion system must be able to handle a wide variety of field measurement units and unit spelling variations.

Process measurements typically belong to one of two unit systems, English or Metric (SI).  Unfortunately, there are multiple units of measurement for some dimensions in each of these systems.  For instance, users of the metric system may use pressure measurements in bar, kilogram per centimeter squared (kg/cm2), or the official standard units of Pascals.  Furthermore, field measurements are typically scaled so that a board operator does see more than five (5) digits in front of a decimal point.  For instance, small volumetric flows in a process unit may be reported of cubic feet per hour (ft3/hr) where as large volumetric flows in the same process unit may be reported as million cubic feet per day (MMft3/day).  The impact of this variety of choices is that engineering unit conversion cannot be performed simply on the basis of whether the end-user’s process field instrumentation is based on the English or Metric system.  Unit conversion must be made based on the field units of a given measurement and the required input units of an engineering calculation.

2.1        Unit Conversion Methodology

As stated earlier, unit conversion calculations are very straightforward, requiring only multiplier and offset coefficients.  The design challenge for unit conversion is to create an efficient methodology that can be applied to a wide number of potential unit conversion choices while keeping it simple for developers and end-users to add new conversions that are not already part of the Engineering Unit Conversion module.

An additional concern for the module is to highlight missing unit conversion definitions as they are requested by engineering calculations.  This will assist the developers and end-users with any necessary customizations.

2.2        Packaged Unit Conversions

3.0        Detailed Design Specifications

The conversion of engineering units will be performed through a single API method supported by pre-defined and user-defined unit conversion objects.  A combination of organization and execution techniques will be used to allow maximum flexibility for the application developer while executing each unit conversion as efficiently as possible.

3.1        Unit Conversion Methodology

The methodology for engineering unit conversion can be separated into two functionalities, the calculation for performing a specific unit conversion and the efficient processing of a set of unit conversions for a specific application.

3.1.1        Unit Conversion Equations

Engineering unit conversions are always a simple linear equation in the form of:

y = Ax        Multiply Only        (1)

or

y = x + B        Offset Only        (2)

or

y = Ax + B        Multiply First        (3)

or

y = A(x + B)        Offset First        (4)

where:

A  - Conversion multiplier coefficient (y units / x units)

B  - Conversion offset coefficient (x units)

x                - Engineering unit to be converted

y                - Converted engineering unit

For unit conversions where either the multiplier coefficient equals 1.0 or the offset coefficient equals 0.0, either equation 1 or 2 can be used for conversion.

For unit conversions where multiplier coefficient does not equal 1.0 and the offset coefficient does not equal 0.0, the order of the conversion calculation is specific and the appropriate equation (3 or 4) must be selected for a given unit conversion.  For instance, conversion of temperature from degrees C to degrees F must use equation 3, whereas temperature conversion from degrees F to degrees C would require equation 4.

Unit conversion objects, however, are intended to be usable in both conversion directions.  A unit conversion object defined to convert degrees C to degrees F will also be used to convert degrees F to degrees C.  For reverse conversions, the inverses of equations 1 through 4 must be used:

Equation 1 inverse is:

y = x / A        (5)

Equation 2 inverse is:

y = x – B        (6)

Equation 3 inverse is:

y = (x – B) / A        (7)

Equation 4 inverse is:

y = (x / A) - B        (8)

3.1.3        Unit Conversion Execution

The end-user will request a unit conversion through a single API method, regardless of the dimension to be converted.  Efficient execution of the unit conversion method will be achieved by organizing the engineering unit conversion objects by dimension to be converted and by prioritizing the search based on the request frequency (future) for each unit conversion.

3.1.3.1        Conversion Request Method

A unit conversion request will be made through a method call in the following format:

y, status = f(dimension-class, x, input units, output units)        (9)

where:

y        - Converted process value, float

status- Conversion status, symbol

dimension-class- dimension to be converted, symbol

x        - Process value to be converted, float

input units        - input x engineering units, text

output units        - output y engineering units, text

This method will locate the appropriate unit conversion object, determine whether the conversion is direct or inverse, and perform the specified conversion equation (1 - 8) using the supplied coefficients.

Once a unit conversion object with the requested input and output engineering units has been located and a conversion direction has been determined, the following logic should be used to select the appropriate conversion equation.

when the equation type of the unit conversion object is MULTIPLY-ONLY:

and the conversion direction is FORWARD

then use Equation (1) for the conversion

otherwise use Equation (5) for the conversion

when the equation type of the unit conversion object is OFFSET-ONLY:

and the conversion direction is FORWARD

then use Equation (2) for the conversion

otherwise use Equation (6) for the conversion

when the equation type of the unit conversion object is MULTIPLY-FIRST:

and the conversion direction is FORWARD

then use Equation (3) for the conversion

otherwise use Equation (7) for the conversion

when the equation type of the unit conversion object is OFFSET-FIRST:

and the conversion direction is FORWARD

then use Equation (4) for the conversion

otherwise use Equation (8) for the conversion

The status returned by the unit conversion method should communicate any problems encountered during the execution of the desired unit conversion.  The following probable status codes are suggested:

DIMENSION-UNKNOWN

An invalid dimension argument was given

INPUT-UNITS-UNKNOWN

The input units argument could not be matched with any existing unit conversion objects (objects in Missing object list excluded)

OUTPUT-UNITS-UNKNOWN

The output units argument could not be matched with any existing unit conversion objects (objects in Missing object list excluded)

MISSING-COEFFICIENTS

The matching unit conversion object is in the Missing object list

ZERO-MULTIPLIER

A conversion multiplier of 0.0 was supplied by the unit conversion object

SUCCESSFUL

Unit conversion was successfully calculated

3.1.3.2        Locating a Requested Conversion

Fulfilling a unit conversion request requires finding the appropriate unit conversion definition object.  The location search is complicated, as it requires the translation of the end-user naming conventions used within their DCS or process information systems to the internal keys used by the unit conversion objects.

Users often have different methods of referring to common engineering units within their DCS and Process Information systems.  For instance, temperature may be labeled as “DEG F” or “DEGF”.  A flow may be labeled as “M3/H” or “m3/h” or “M3 / H” or “m3 / h”.  Often, multiple versions will be used within a single DCS or Process Information system.  Therefore, all input and output request and unit conversion object input and output specifications will be stripped of spaces and converted to all capital letters during the search process to minimize the number of unit alternate unit spellings that will need to be defined for a given unit conversion object.  This will allow a unit request for “m3 / h” to be matched with a unit conversion object defined to convert “m3/h” as both unit specifications would be converted to “M3/H” during run-time execution.

There are two distinct functions required to locate the appropriate unit conversion definition object for a given request.  First, the conversion synonym objects for two unit key parameters of each possible unit conversion definition object for the specified dimension class must be found.  Second, the contents of the synonyms of the input and output unit keys of each unit conversion definition object must be compared with the requested input and output units.

3.1.3.2.1        Associating Synonyms for Unit Keys

A single Conversion Key Synonym Definition Object can be used for multiple Unit Conversion Definition Objects.  Therefore, it is desirable to relate each unit conversion object with its two required synonym definition objects.  Creating the relations is typically performed when creating or modifying a new unit conversion object.  During unit conversion validation after creation or modification using the user dialog, a search is made of all available synonym definition objects to match the input and output keys.  The following G2 relations are created to associate the appropriate synonym definition objects to the unit conversion object:

Relation

 

the-input-key-synonyms-for

Synonyms for the Input-Key-Text

the-output-key-synonyms-for

Synonyms for the Output-Key-Text

3.1.3.2.2        Matching the Requested Units

Unit conversions are bi-directional in that the coefficients used to convert the defined input units to the defined output units can also be used to convert the output units back to the input units.  Therefore, it is necessary to specify any give conversion for only one direction, and the matching of user requested units to a unit conversion object requires checking each unit conversion object for both forward and reverse conversions.  An internal direction status is set based on the results of the unit conversion object matching.

if the requested input units

match a synonym of the input-key-text of a unit conversion object

and the requested output units

match a synonym of the output-key-text of a unit conversion object:

then the conversion direction is FORWARD

if the requested input units

match a synonym of the output-key-text of a unit conversion object:

and the requested output units

match a synonym of the input-key-text of a unit conversion object

then the conversion direction is REVERSE

if the requested input units

match a synonym of the input-key-text of a unit conversion object:

and the requested output units

match a synonym of the input-key-text of a unit conversion object

then the conversion direction is NO-CONVERSION-NEEDED

if the requested input units

match a synonym of the output-key-text of a unit conversion object:

and the requested output units

match a synonym of the output-key-text of a unit conversion object

then the conversion direction is NO-CONVERSION-NEEDED

3.1.3.3        Conversion Object Organization

It is anticipated that over time that there will easily be hundreds of potential unit conversion objects that can be accessed by an application developer.  Therefore, an efficient method of finding a specific unit conversion object during run-time is critical.  One method of improving the efficiency of finding a specific unit conversion object is to organize all unit conversion objects into dimension-class specific lists.

During the initialization of an application, a unit conversion organizer object will be created to hold a data structure the will be used to facilitate the location of a desired unit conversion object for a given unit conversion API request.  The organizer object will have a single attribute:

EUCD Structure

Structure

Unit conversion organizer structure

There are to be three sub-structures within the EUCD structure attribute:

Used-EUCD

Structure

Used unit conversion object structure

Unused-EUCD

Structure

Unused unit conversion object structure

Undefined-EUCD

Structure

Undefined unit conversion object structure

3.1.3.3.1        Unused Object Structure

During application initialization, all existing unit conversion objects are to be placed into an Unused object structure, which will contain a series of dimension-class specific sequences of unit conversion objects.  When a specific unit conversion request via the API procedure call is matched with a unit conversion object in the Unused object structure, the object will be removed from the Unused object structure and placed into the Used object structure.

3.1.3.3.2        Undefined Object Structure

The Undefined object structure is initially empty.  Whenever a specific unit conversion has been requested via the API procedure call, but no matching unit conversion object can be found in either the Used, Unused, or Undefined object structures, a new unit conversion object will be created.  The following unit conversion object attributes for the new object will be modified from the default values and set using the parameters passed by the API call:

Dimension Class

API dimension class

Input Key Text

API input units

Input Texts

Sequence (API input units)

Output Key Text

API output units

Output Texts

Sequence (API output units)

The Undefined object structure is intended to consist of one or more dimension class specific sequences of unit conversion objects.  The newly created unit conversion object will be inserted into a dimension class specific sequence and a alert message sent to the application developer that an undefined unit conversion request has been made.

The developer, upon receiving an alert message, can take one of two actions.  First, if a mistake was made in the API call, it can be repaired and the new undefined unit conversion object can be manually deleted via a unit conversion manager.  Second, if the API call is correct, the developer can edit the new unit conversion object, supply the appropriate conversion parameters, and save the modified object via a unit conversion manager and dialog.  The modified unit conversion object is checked to make sure that is does not conflict with any existing unit conversion objects.  If the unit keys are unique, then the Confirmation-Status attribute is set to true and the object is move to the appropriate dimension class sequence in the Unused object structure.

3.1.3.3.3        Used Object Structure

The Used object structure is a series of dimension class sequences of unit conversion objects for which at least one previous API request has been made by the containing application.  Any unit conversion object in the Used object structure will remain in the structure until the application is restarted or the developer modifies the unit conversion object parameters.  Any newly modified unit conversion object should be re-validated and moved into either the Unused or Undefined object structures.

3.1.3.3.4        Object List Search Technique

When a user unit conversion API request is made, the appropriate unit conversion object will be located using the following search technique:

  1. 1.The Used object structure dimension class specific sequence 

  2. 2.The Unused object structure dimension class specific sequence 

  3. 3.The Undefined object structure dimension class specific sequence 

As specified in section section 3.1.3.2, each unit conversion object in an object sequence should be checked against the desired input and output units for both conversion directions so that it is not necessary for the developer to define a conversion and its inverse.  For instance, if a request is made to convert degrees F to degrees C, the applicable unit conversion object may be specified in the orientation of converting degrees C to degrees F.  Therefore, each unit conversion object should be checked to see if the inverse of the specified unit conversion will provide the requested conversion.

If the unit conversion object that matches the requested units is currently in the Undefined object structrure, then the API method should return an unsuccessful status value.

This search technique should, within a few hours of application initialization, move most, if not all, of the necessary unit conversion objects for the application into short, dimension specific object sequences within the Used object structure.  While initial unit conversion API requests may take some time to process, subsequent requests should be executed very efficiently.

3.1.3.4        Conversion Request Frequency (future)

As stated previously, the search for the appropriate unit conversion object for a requested unit conversion is somewhat cumbersome.  As a further refinement to the search technique specified in Section 3.1.3.3, the order of the unit conversion objects within the Used object structure dimension class specific sequences will be updated based on the frequency of conversion requests.  This will improve unit conversion search efficiency by checking the most heavily used unit conversion objects within a dimension class specific sequence first.

Each Used object structure dimension class specific object sequence is to be periodically scanned for the need to re-order the unit conversion objects.  A time-based algorithm is to be used to initiate the potential re-ordering of Used sequences.  The algorithm will initiate the re-ordering of the Used object structure sequences based upon the time since application startup.  The need for re-ordering the Used object structure should decrease as the time since application startup grows, since it is expected that the access order for each sequence will stabilize within a few hours of application startup.

3.2.2        Conversion Key Synonym Definition Object Files

3.2.2.1        Gensym Packaged Synonym Definitions

Pre-built conversion key synonym definitions are to be supplied for each unit key required for the Gensym packaged conversion definitions.  The synonym data specifications will be managed using Excel spreadsheets that are converted to CSV files for loading the Engineering Unit Conversion module.  The spreadsheets should be updated with new synonym definitions as application developers find the need for additional synonyms or as new unit keys are specified for the packaged conversions.

When creating custom unit key definitions via the custom loading file, each new definition must be validated to insure that all required information is provided and to prevent duplication of existing unit key definitions

The custom synonyms are loaded into the Custom-Texts sequence attribute of the appropriate Conversion Key Synonym Object.  The values in the Custom-Texts attribute may be modified at any time by the developer or end-user via the user dialogs.
