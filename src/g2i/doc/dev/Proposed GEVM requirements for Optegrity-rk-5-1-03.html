Proposed GEVM requirements for Optegrity/SymCure

 

The goal of this document is to identify what capabilities we need from the message management module GEVM both for now and the future.  The underlying assumption is that GEVM will ultimately take over all message management requirements not just for Optegrity and SymCure (in the near term) but also for Integrity (in the not so near, but hopefully not too far, future).

 

Messages and their containers.

 

  1. 1.Support for multiple message container queues.  GEVM by default sends all messages to a single queue.  Providing multiple queues will provide developers with the flexibility to organize messages in ways not supported by browser level filtering.  Note that a message may exist in multiple containers at the same time. 

 

GEVM has a main queue for operator messages and another for raw events where all messages go into. But this does not stop anybody to add events to other queue as well, by calling the appropriate API GEVM-ADD-EVENT-TO-QUEUE. Also if I look at the C&W solution or the Aramco solution they dump the messages into a spreadsheet in VB. Well the least we can say is that we are dealing with different camps (sounds like Java versus .NET versus …king of dilemma).

 

The current setup in our development environment is to dump all messages into one queue including the ones from CDG (and display a type column with each message such as root-cause, incoming event having their own type to be able to differentiate them). So what do you think about the following: By default we have one message queue in the server with all messages, i.e. basic operator messages and CDG messages. Then in the CDG event listener that transforms notification events from CDG to operator messages, we get the name of the desired target queue from the config.txt file (if none is defined then all messages go to the main queue). We would define four keywords in the config.txt file that we will lookup for the target queue name for CDG events. If the queue does not exist then we dynamically create it and store it in the default repository transparently to the user and then store any messages in this queue instead of the basic queue. I will change the APIs to ensure that will happen, by creating a new set of create-event APIs where you can pass the target queue as well. The existing APIs will send the message to the current main queue.

 

The user or application builder can then decide if he wants one or multiple queues by configuring it in the config.txt file. We take care of the coding. We create the menus to browse the queues dynamically based on the current instances of the queues (except the one the palette of course). We can then create a root queue class for raw events/black board and another for operator messages.

 

If we agree, I take care of this. Ravi also pointed out that the ui-client-item of the gevm APIs should be removed (came from earlier practices to always a win in all APIs I guess).

If I understand this correctly, you are saying we can support separate queues for CDG messages where we will be able to send alarms, root causes, tests, and repair actions.  We will send events to these queues will occur by creating them with a new create-gevm-event API.

 

This works for me.

 

  1. 2.It should be possible to specify the container that a message is destined for at the time of creating the message.  If no container is specified, the default behavior may then send the message to a default queue.  This way, we can have the best of both the current GEVM and SMH design. 

 

See above. Solution is to create a new set of gevm-create-event APIs taking one more argument, which is the target queue name. The existing ones will send it to the default queue.

 

  1. 3.A message should have the following attributes: 

    1. a.Sender A relation exist between sender and event/message with APIs to access it. If you display something in the message (attributes) then if the sender key or label changes we need to update the attribute on the message and it uses extra space. Displaying something in the message browser (and collect the text dynamically would be more efficient), but what would you want to display? The item key? The browser also has the show initiator to view it – isn’t it enough? 

    2. b.Target Same as above but for target 

    3. c.Category (GEVM defines this to be a symbol.  However it would better to use a text for display purposes.) 

    4. d.Repetitions Exists in GEVM 

    5. e.Creation time Exists in GEVM 

    6. f.Message text (e.g., the name of an event) Exists in GEVM – gevm-message attribute 

    7. g.Additional text (e.g., the value of the event) Exists in GEVM – gevm-detail attribute 

    8. h.Still more text (e.g., the status of the event or allow subclassing of message for SymCure’s diagnosis console) GEVM has the advice attribute or you can subclass it. Couldn’t this text be added to the detail? 

 

If you do as suggested in point 4, Message text and additional text will be sufficient for displaying status and value of an event separately.  (Point 4 proposes that category be made into a text instead of a symbol.)

 

    1. i.Time sent Exists in GEVM – gevm-creation-timestamp attribute 

    2. j.Priority Exists in GEVM – gevm-priority attribute 

    3. k.Lifetime Exists in GEVM- gevm-lifetime attribute 

    4. l.Acknowledgement status Exists in GEVM – gevm-acknowledged attribute 

    5. m.Acknowledger Exists in GEVM – gevm-acknowledged-by-user attribute 

    6. n.User comment Exists in GEVM – gevm-comments attribute 

 

SMH also provides the following additional attributes which may be considered for inclusion:

    1. o.Formatted creation time Not explicitly included in GEVM messages to save memory space. The time stamp is dynamically converted into a textual format when displayed in browser. 

    2. p.Revisit time  gevm-last-update-timestamp attribute and in timestamp format that can be converted to text as needed. 

    3. q.Deletion time Not included currently. Would you want to keep the message around while it is flagged to be deleted or delete it immediately? 

 

  1. 4.SymCure requires that messages be uniquely identified by a combination of the target and a text attribute.  (In SMH, target and category are used.) 

 

GEVM uses the following to build the key of a message. Msg Key = "[the name of EventClass].[initiator-key].[target-key].[category]";

 

Is EventClass the class of the gevm-event, e.g., gevm-advisory, gevm-cdg-event?  If this is so, this MsgKey is not suitable for uniquely identifying SymCure events.  I recognize that you have used the category attribute to represent the SymCure event name in g2i-dev but this is problematic because SymCure event names are texts not symbols.  In CDGUI, I have also used the category attribute for the SymCure event name, but because SMH uses a text for a category, it’s an acceptable situation.

Perhaps the simplest way out is to make category a text attribute.

 

  1. 5.When a message is repeated, it should be possible to replace the message text in the queue, not just increment its count.  SMH achieves this by providing an option in the API smh-create-message which specifies whether to simply increment the message count or to replace the message text in the case that the message already exists.   

 

Which ones of the above list would you want to update? Category, Message, detail, advice, priority, lifetime? If category is changed would you want the key to be updated as well? Would you reset the creation timestamp or only update the last update timestamp? This option would be added to the gevm-create-event API

 

In general, I think it should be all the non-key attributes except the creation timestamp, but in particular for SymCure I need to be able to update Message, detail, and/or advice. (This is assuming category becomes a text that will contain the event name.)

 

  1. 6.Provide APIs to delete a message in a specified container, to delete a message in all containers, and to delete all messages in a container.  As part of the API or as part of the functionality of the API, there should be a way to delete a message without having to acknowledge it first. 

 

I suppose you mean remove from a specific container (via delete APIs on queue for example) and only really delete the event/message (g2 delete) if it is not included in any queue anymore. PrOpEx enforced that you had to ack a message before you could delete it. Can you delete a message in SMH before you ack it?

 

In situations where acknowledgement has no meaning in the context of the display, e.g., SymCure’s events and external actions that are shown on the diagnosis console, you should be able to delete without acknowledging.  In the SMH implementation, these messages are deleted by SymCure through APIs once the underlying events are deleted.  And they are never acknowledged, so SMH does allow you to delete a message before you aknowledge it.

 

  1. 7.Provide API’s to acknowledge a single or multiple messages in a specified container, acknowledge a single or multiple messages in all containers, and acknowledge all messages in a container. 

 

I take it as an API to ack multiple messages but once you ack a message it will be acked in all queues (no ack state per message per queue it is a member of). Yes for the API to ack all messages. I don’t understand acknowledge a single or multiple messages in all containers which would mean you can ack a message in one queue but not in another it is part of. How is this currently used?

 

  1. 8.Messages and Containers must be subclass-able, allowing the Customer to extend its default attributes 

 

As long as they are subclass of gevm-event for raw events and gevm-message for operator messages

 

Message browser

 

  1. 1.A message browser must be able to subscribe to one or more containers and display their messages.
     

Why? How do you differentiate them in the browser? What not a queue that listens to the others and has the union? This is not clear to me why this is important (and maybe confusing with so many options) compared to the number of options we already have with multiple queues / single queue, messages that can be in multiple queues, and filters at the ui.

 

  1. 2.It should be possible for an internal or external developer to create a customized browser. This may require dialogs to create columns, configure their dimensions, specify message attributes to display in the columns, and provide callback procedures to dynamically assign colors to the cells based on the properties of the message.  (Currently all this is possible in GEVM through tables.) 

 

GQS enables to create custom browsers and also popup a different browser depending on user and user-modeSee the configure tabular view on gqsv-tabular-view-template instances. In our ReThink client (using GEVM) this is done in the browser – If you want to I can have Poshika show it to you.

 

  1. 3.A typical message browser is likely to require the following buttons: 

    1. a.Acknowledge – Exists in gevm default browser 

    2. b.Delete – Exists in gevm default browser 

    3. c.Go to sender – Exists in gevm default browser 

    4. d.Go to target – Exists in gevm default browser 

    5. e.View message details (very important unless we can dynamically adjust column widths!) – This has actually two parts and none exists currently in gevm: 1. a custom button to launch the properties dialog of a event/message and a guide dialog for the event/message 

    6. f.Add comment – to be done 

    7. g.Sort by any viewable column. – Seems to work in gevm 

    8. h.Reverse the sort order – Seems to work in gevm 

    9. i.Lock or unlock view – Seems to work in gevm 

 

It should be possible to create browsers with customized buttons (e.g., SymCure’s diagnostic console). GQS supports it and the go to source and target of the default message browser are custom buttons. I guess we will also need to support the custom buttons you added to the CDG consoles as well, which should not be that hard I support, correct?

 

  1. 4.Message browsers must include the following kinds of filters: 

    1. a.Target filter: show all messages on Target and the objects on its subworkspaces – To be done 

    2. b.Priority filter: show all messages of Priority X – Do you want to the user to specify a specific priority or buttons for each priority? 

    3. c.Must have the ability to filter based on any attribute of a message. – configuration dialog exists in default browser 

    4. d.Need to support colors for messages based on their priority 

There should be a callback procedure that by default will use priority to assign a color to the message but can be replaced with something else depending on the browser.  For example, in SymCure’s SMH-based diagnosis console for alarms and root causes, I use the event value and status to determine the color for the message.  This makes the colors of the events consistent with the colors on the Specific Fault Propagation Models.  This is critical because the diagnosis console is basically a view of the SFPM.  But this is easily done with existing capability --- look at the dynamic-color-formatter attribute for a GQS column.
