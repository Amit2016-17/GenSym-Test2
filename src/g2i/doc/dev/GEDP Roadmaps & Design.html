Overview

Introduction

This purpose of this G2i plug-in module is to provide a generic and high performance graphical language for G2 based applications. Compared to other graphical languages such as GDA, it should stay at its core purpose for event and data processing and not include other functionality such as alarm management.

 

As with all G2i modules, GEDP may rely on the core G2i modules, but should not require other plug-in modules. The final application or product will drive the need for one or more plug-in modules. Similar to the other modules, GEDP should support the Meta model defined in GRTL as appropriate.

Purpose

The purpose of this document is to list the requirements for the plug-in module including a phased in delivery approach and describe key design issues and implementation issues.  The intended audience for this document includes the following:

  • .potential users of the product, such as consultants, solution engineers, and end users; 

  • .managers involved in the marketing or management of the product;  

  • .people with experience in the target markets, such as strategic sales;  

  • .technical writers responsible for documentation; 

  • .trainers responsible for developing courses for the product; and  

  • .other product developers in need to understand the product evolution and key design issues

General Requirements

Software Environment

GEDP needs to run all G2 supported operating systems. This should not be any problem since it is implemented at the KB level.

Scalability and Performance

The GEDP module needs to be designed for high performance and to outperform current graphical languages such as GDA for data processing.

Security

None

Compatibility / Migration

None

International

The GEDP module needs to support internationalization for all user and error messages.

Licensing

No specific requirement. It needs to be licensed similar to all G2i modules.

Bundle Inclusion

This module should be part of and integrated with the following bundles: Optegrity.

Summary

ODP lacks a significant amount of the functionality available in GDA but has several features which make it very suitable for this role.  GDA is a much more mature product with generally more functionality.  ODP has features which allow it to more easily interface with some other application products.

 

The functionality advantages which GDA offers in its present state are:   Override capability, Charting capability, Fuzzy logic, SPC, Custom blocks, a Reset capability, Multiple invocation options, an Explanation facility, Alarms, and Variables and Parameters (VorPs) in diagrams.  It also features the ability of some blocks to handle synchronization of inputs where all or some specific inputs must receive a new value before the block can fire or where an input on one path causes the block to wait for an input on another path.

 

While ODP is missing the functionality listed above and also lacks some of the block and diagram protocols that GDA has, this can also be seen as an advantage for ODP.  It is not tied to a queue implementation and the protocols that it lacks may make it more flexible for fitting to future or newly existing (G2i) products and toolsets.  ODP is also (currently) much more lightweight than GDA.

 

The functionality that ODP lacks can be added in the future and some of the functionality that GDA has is probably not needed.  Tabular gates, for instance, are hardly ever used and were developed in the early days of GDA for one customer.  GDA has, over the years, become somewhat rigid in spots and where different functionality is required, it would be easier to build it into ODP than change or remove it from GDA, particularly where GDA has its own customer base that relies on the current implementation.  Changing GDA to please one set of customers may well annoy another set.

 

In the long (and perhaps the short) run, we would probably do better enhancing ODP than trying to hammer GDA into a workable configuration.

Design Overview

Design issue notes

 

  • .The update-rendering procedure needs to have an Off switch somewhere. 

  • .There needs to be some sort of allow-other-processing within the execution scheme. GEDP can cause major delays in other G2 processing. 

  • .To make the block evaluators UI-free requires a core execution routine that does not refer to input paths just input values and items. 

  • .No fuzzy logic support exists at this point 

  • .What about inference or value filtering? 

  • .The standard-deviation block requires an initialization after it is configured to make it run. 

  • .Block requiring an upstream float-variable block for history do nothing if the float-variable block is not configured to keep a history. Needs to be described in documentation 

  • .There should be a junction block to handle paths joins rather than allowing ad hoc junctions, otherwise the evaluator has to waste time tracking down which block attached to an input actually caused the evaluation and which upstream block’s output to get. 

  • .There are existence checks in some evaluators or procedures called by evaluators which are overhead 

  • .Does anything call _gedp-fetch-upstream-port-name? 

  • .Add sample-period to sine-wave-generator. 

  • .39 object-definitions have the class specific attribute gedp-value-attribute-name. 

  • .Does the float-variable block, when is it used as an upstream history holder, actually provide anything useful?  Why not just add a variable to the downstream block to hold the history and dispense with the upstream variable?  See Item #11 above. 

  • .A generalization derived on Item #23:  No block evaluator should be dependent on a proceding block or the attributes thereof (ie. once a block has evaluated, it is done and the next block evaluates).  Any data required by the downstream block should be stored in the output of the upstream block.  In GDA only 2 blocks have locked dependencies on the upstream block and that is so that they may handle a varying number of inputs easily; otherwise a complex configuration would be required. 

  • .Should there be capability objects similar to those in GDA? 

  • .Need to move master diagrams to non-proprietary workspaces. 

  • .The evaluator propagates either a token (currently a sequence) containing a structure with data & collection time of that data or it propagates an item (actually the symbolic name of an item).  This practice has to be standardized.  The token should always be propagated. Additionally the token, which should be transparent to the evaluator or its proxies is not.  The carrier seems to get in the way of data access when trying to deal with objects. 

  • .We need to enforce connection limits.  Currently it is possible to connect an output path into the body of a GEDP block (at least most of them) not at the input stub, propagate a value on that path and have the block execute normally.  This is very problematic and must be prohibited.  Path splits are perfectly OK but path joins present all sorts of problems for an explanation facility and should be outlawed. 

  • .We will be creating more multiple input, non-peer input blocks in the future.  The current arrangement of  passing a pile of arguments to the block evaluator (or receive signal routine) works well for single-input blocks but is going to hamper us more as we push on with GEDP development.  Blocks should be going after their inputs (via the relations)  and not relying on the upstream signaling block to inform them of the evaluation details. We will constantly end up with a conditional; as an example:  if got input 1 then go get input 2 else go get input 1.  The example is only for a 2-input block; if we have 3 or more inputs its messier. 

  • .Peer input blocks should be a mixin class. 

  • .The relations used to create the virtual diagram should not have inverses; there should be 2 non-inverse relations formed to ease maintainability.  The upstream block does not care whether a particular output port is connected to an input-1, input-2, etc.  The block just wants the outputs for that port.  The downstream block on the other hand is very concerned with which port an input comes into.  For single-input and peer-input blocks, currently the vast majority of GEDP blocks, the second relation is not even needed. Alternatively we could create a function or procedure which would return a list of  the downstream blocks. 

  • .The threshold connection relation should be replaced with a normal input relation.  There are many other “special” connections we could relate but it complicates the process.  There is no reason why the threshold input needs to be different from any other input. 

  • .If we wish to develop the functionality of diagrams without actually building a diagram then we should standardize the Receive, Evaluate & Signal methods to insulate the evaluator from the physical block as much as possible.  Specifically, the Receiver should unpack the input and spoon-feed the evaluator. Done properly, the evaluator would not require any “knowledge” of the physical block at all.
