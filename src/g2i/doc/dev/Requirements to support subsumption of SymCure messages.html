Requirements for subsuming of messages generated for SymCure Events

 

GEVM provides the functionality for a message to be subsumed by one other message. The goal of this document is to see how this capability can be used and possibly extended to subsume messages that arise from SymCure events and actions.

Motivation

Motivation : Also to reduce primary number of messages and organize them for the operator to let him focus on what’s imp

 

The operator message browser displays messages about SymCure’s events and actions.  A message is generated for any event that is configured to be a root cause or an alarm whenever the value or the statue of the event is changed.  Likewise a message is generated for any action whenever its status is changed.  The motivation for subsuming SymCure messages is to assist equipment operators by replacing a potentially large number of alarm messages with a concise set of root cause messages.  

Requirements

1.        Message subsuming provides a tabular view of the summary of a root cause event.

The summary view for a specific root cause event summarizes the key relationships between the event, its causes, effects and actions, without displaying the complete details of the specific fault model.  The goal of message subsuming is to reproduce the summary view for a root cause in tabular form under the message for the root cause in the message browser.  

 

For example, the figure above shows a root cause “Impure catalyst” with known alarm (i.e., symptom) “Retarded chemical reaction”, predicted alarms “Low ROI”, “Impure product”, and “Wasted reagent” and actions “Impure catalyst?”, “Purify catalyst”, and “Raise temperature setting”.   If message subsuming were in effect, then any messages about the symptoms, predictions, and actions would be hidden under the message about “Impure catalyst”.

 

2.        The subsuming relationship between messages is many-to-many.

Note that an event may have multiple root causes and an action may be associated with multiple events.  In the same example, “Primary supply failure” is another root cause which shares some symptoms, predictions, and actions with “Impure catalyst” (see figure below).

 

 

Therefore, it must be possible to support a many-to-many relationship between a parent message and its subsumed messages.  (Presently, the GEVM relation the-parent-event-of is a one-to-many relation.)

 

3. The behavior of subsumed messages is dynamic and non-monotonic.

Causal relationships between specific events depend on the values of the events.  In the figures above, suspected root causes were treated as possible explanations for symptoms and predictions.  In the same example, the knowledge that “Inadequate heating” is false, changes the summary view of “Primary supply failure” as shown below.  “Primary supply failure” now inferred to be false can no longer be regarded as a potential explanation for “Retarded chemical reaction”, “Low ROI”, “Impure product”, and “Wasted reagent”.

 

 

Thus, any messages about events once subsumed by “Primary supply failure” must no longer be subsumed by it.  

 

4. Only one level of subsuming is sufficient for tabulating the summary view.  

The summary view directly relates symptoms and predictions to a root cause without regard to the intermediate events and the structure of the specific fault model.  Thus, it does not require a cascade of subsuming (i.e., the-parent-event-of) relations.  This should be adequate; several graphical views of the structure of the specific fault model are already available.

 

5. The option to subsume messages about SymCure’s events should be provided separately.

When it comes to messages about SymCure events, the operator message browser behaves like the SymCure diagnostic console and therefore disallows acknowledgement and deletion.  Since it treats messages about SymCure events differently, it might be beneficial to allow separate configuration option for subsuming these events.
