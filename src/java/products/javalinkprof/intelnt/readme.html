G2 JavaLink Professional

G2 JavaLink/Professional provides;
  • Import of Java class definitions into G2.
    • An imported Java Class maintains its class inheritance when imported into G2, as JavaLink Professional will automatically import any classes and interfaces found from the Java Classes inheritance definition.
    • Method stubs are created for every public method and constructor found using reflection. JavaLink Professional will automatically import any Java Class definition required for classes used in method and constructor signatures.
    • Imported Java Constructors can be called to create a G2 "proxy" for an imported Java Class. An associated Java object associated with the proxy can then be controlled from G2 by calling the imported method stubs.
  • Support for all Java types including arrays.
    • G2 JavaLink/Professional complements G2 JavaLink/Gateway by providing additional classes that allow all types and arrays to be sent between G2 and Java visa versa.
  • G2 Runtime Export
    • This very powerful feature enables a JavaLink Professional user to export G2 classes and method behavior from G2 to the Java environment. JavaLink Professional implements a G2-EXPORT-SERVICE that allows fundamental G2 foundation classes to be exported into a running Java VM. JavaLink Professional also provides an implementation of a G2-RUNTIME that runs on a Java VM, so that exported G2 classes can be created and controlled. This release allows G2 CLASS-DEFINITIONs and their methods to be exported. Future releases will allow other built classes to be exported (such as RULES and FUNCTIONS, etc).
    • G2 methods are exported  using the G2Script Compiler ( See G2Sscript Description ). G2Script is a powerful subset of the G2 Language specification, that can run within a Java VM.

Setting Environmental Variables

    In order for G2 JavaLink Professional to function, additions must be made to the machines  CLASSPATH environmental variable.

    While the InstallShield installation program is capable of make the necessary changes to the environmental variables, this facility may be vetoed by the installer.

    Note: The following instructions apply only if you elect not to allow the JavaLink Professional install shield process to make JavaLink Professional specific changes to the machines  CLASSPATH environmental variable for you. Also, if you allow the JavaLink Professional install shield process to perform the step for you (the default) then you must reboot or logout/login for the changes to take effect.

    The following assumes that products have been installed in their default location. Modifications should be made where appropriate when products have been installed in an alternative directory.  In addition it assumes that PATH and CLASSPATH have already been set to reference both the JDK and G2 Javalink/Gateway.

         
  • Your  CLASSPATH must be setup to point to the G2 JavaLink Professional  files.
    • To setup your current NT session for all applications you must use the system icon on the NT control panel. Select the ENVIRONMENT tab and enter the following.
      • variable JAVALINKPROF_HOME, value (installation directory e.g. c:\gensym\javalinkprof)
      • variable CLASSPATH, add the following ;%JAVALINKPROF_HOME%\classes;%JAVALINKPROF_HOME%\classes\javalinkprof.jar;%JAVALINKPROF_HOME%\classes\antlr.jar

        •  
    Please note, if you have Gensym's Telewindows2 Tookit installed, please ensure that JavaLink Professional's entry on the CLASSPATH comes before Telewindows2 Toolkit classes entry on the CLASSPATH.

    Problem Solving

    General

    (please refer to the current know problems/bugs list in KnownProblems.html)

    Memory Heap Size

    The Java interpreter utilizes a memory heap for all object references and values. When the heap becomes full, the interpreter will not try to allocate any more memory, but will throw an OutOfMemoryException. If Java throws an OutOfMemoryException while running a Java application or applet, then try the Java interpreter's -ms option. e.g. java -mx20m classname will increase the Java heap size from its default (currently 16meg) to 20meg.

    Java Thread Memory usage

    Java on the NT platform creates a  native NT thread for each java.lang.Thread that is started. We have found that a significant amount of memory is allocated for each thread that is created. Additionally, thread memory is not allocated from the Java heap, but allocated from the overall processed memory.
     

    Problems when using Symantec's  JIT supplied with the JDK1.1.6

    We have not seen any significant problems when testing JavaLink with the Symantec Just In Time Compiler (JIT) supplied (and turned on by default) with JDK1.1.6. However, if you experience any unstability we would first recommend that you run the JDK VM with the JIT turned off by using java -nojit classname, and then retesting.
